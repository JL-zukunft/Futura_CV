# 案例文件MD格式规范

## 1. 文件结构

每个案例文件应遵循以下结构：

```markdown
---
id: case-01              # 案例唯一标识符
caseNumber: 1            # 案例编号
title: AI 内容策略引擎  # 案例标题
projectName: Content Strategy AI  # 项目名称
mainTitle: 当一个品牌每周需要<br/><span class="text-futura-green underline decoration-2 underline-offset-8">100 条文案</span>时  # 主标题（支持HTML标签）
challenge: 品牌团队内部内容需求持续增长...  # 挑战描述
approach: 建立基于品牌资产库的 AI Agent 生产集群...  # 解决方案概述
frameworkTitle: 问题解构与能力映射  # 框架部分标题
---

## 问题解构与能力映射

### 01. 拆解 - 业务场景拆解
识别 RED/Tmall 调性差异，量化文本特征。

### 02. 识别 - AI 节点识别
确定替代环节与人工审核边界。

### 03. 连接 - 流程连接测试
打通 RAG 知识库供给，避免模型幻觉。

### 04. 封装 - 工程化封装
沉淀 SOP 交付物，实现运营自主产出。

## 核心实施方案

### Prompt 工程精细化：从 "指令" 到 "灵魂"
构建复杂 Prompt 体系，涵盖角色定位、背景约束与结构化输出格式。

**标签**: Role Selection, Context Constraint, Format Control

### 知识库 RAG 增强：拒绝 "通用化"
接入品牌私有数据库，实现文案生成的历史溯源与精准检索，确保每一句话都有品牌积淀。

**标签**: Private Data, Real-time RAG, Brand Consistency

## 项目交付成果

### 10x - Efficiency Gain
文案初稿生成耗时从 45 分钟缩短至 30 秒，显著释放品牌创意人力。

### 92% - Adoption Rate
精准的 RAG 知识库与 Prompt 工程，使人工对 AI 初稿的微调率低于 10%。

### 0.0 - Compliance Error
内置合规检测节点，上线至今未出现一例敏感词或政策性违规问题。
```

## 2. 格式要求

### 2.1 YAML Frontmatter
- 所有字段为必填项
- `id` 必须唯一，格式为 `case-XX`
- `caseNumber` 必须为数字
- `mainTitle` 支持 HTML 标签

### 2.2 主体内容

#### 2.2.1 问题解构与能力映射
- 使用 `## 问题解构与能力映射` 作为标题
- 每个能力项使用 `### XX. 标题 - 子标题` 格式
- 每个能力项后紧跟描述文本

#### 2.2.2 核心实施方案
- 使用 `## 核心实施方案` 作为标题
- 每个方案使用 `### 方案标题` 格式
- 每个方案后紧跟描述文本
- 标签使用 `**标签**: 标签1, 标签2, 标签3` 格式

#### 2.2.3 项目交付成果
- 使用 `## 项目交付成果` 作为标题
- 每个成果使用 `### 数值 - 成果名称` 格式
- 每个成果后紧跟描述文本

## 3. 最佳实践

1. **保持一致性**：使用统一的格式和结构
2. **简洁明了**：描述文本应简洁明了，突出重点
3. **使用 Markdown**：充分利用 Markdown 语法，如粗体、列表等
4. **版本控制**：MD 文件易于进行版本控制，建议使用 Git 管理
5. **可维护性**：结构清晰，便于后续修改和扩展

## 4. 工具支持

- **编辑器**：推荐使用 VS Code、MarkdownPad 等支持 Markdown 的编辑器
- **预览**：使用 Markdown 预览功能查看最终效果
- **转换**：使用 `generate-cases.js` 脚本将 MD 文件转换为 HTML 详情页

## 5. 示例文件

- `case-01.md`：完整的案例文件示例

## 6. 注意事项

- 文件名应与 `id` 保持一致
- 避免使用特殊字符，使用英文和数字
- 保持文件编码为 UTF-8
- 定期备份案例文件
